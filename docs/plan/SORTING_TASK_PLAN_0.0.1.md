#prompt
# 目的

这个微服务负责对象存储服务器目录下的话单文件进行分拣。
服务有一张配置表保存文件服务器相关信息，并且根据配置表信息按步骤进行服务器连通性测试。
微服务提供四个对外接口：
- 第一个接口负责进行文件服务器配置的新增保存
- 第二个接口负责对已有的文件服务器配置修改
- 第三个接口负责触发文件服务器配置连通性测试，当连通性测试通过时刷新配置表。
- 第四个接口控制文件服务器配置启用和关闭

# 上下文信息

配置表包含信息如下：
- 唯一主键 id(系统自动生成uuid)
- 服务器地址（不允许为空）
- 服务器端口（不允许为空）
- 存储桶 Bucket 名称（不允许为空）
- 服务器AK（不允许为空）
- 服务器SK（不允许为空）
- 文件存储服务器目录（不允许为空）
- 服务器连通性是否验证通过（默认不通过）
- 配置是否启用（默认关闭）
# 约束

- 要求文件地址必须是对象存储服务
- 数据库配置表建表语句需要创建，并且保存在代码目录（项目中etc目录下）
- 数据库配置表对ak/sk必须进行安全加密
- 表字段存储均为字符串，要求对某些特殊字符进行过滤，防止渗透攻击
- 连通性测试通过之前禁止启用文件服务器
- 文件服务器配置信息新增修改时，不允许主动修改连通性、启用/停用字段、唯一主键id
- 接口非空入参都需要存在入参校验规则

# 要求

- 接口设计时延要求 500ms。
- 连通性测试成功的服务信息，需要更新配置表对应字段，并且不再重复触发连通性测试。
- 所有接口返回存在异常时，除了 http 状态还需要包含业务错误码。
- 对于异常场景触发时，接口返回需要返回具体的错误信息例如:
~~~
{
	"code":"FAIL.00001",
	"message":"服务器连通失败，服务器地址异常"
}
~~~

- 测试时，不进行实际服务器连接测试，对于所有验证时需要进行服务器连接，均使用Mock 进行打桩进行测试。

# 验证方式

- 需要编写本地 UT 测试用例验证方法正确
- 本地 UT 使用内嵌数据库，创建临时数据库，并执行建表语句
- 验证所有的接口入参校验，以及所有的正确异常返回结果，验证数据库结果是否符合预期
- 异常场景需要包括所有字段的边界校验
