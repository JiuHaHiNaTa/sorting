# 目的

我们已经有file_server_config表来存储文件服务器配置信息，需要通过配置信息连接文件服务器，扫描目标目录下的所有文件。
按步骤进行以下操作来分拣话单：
- 读取所有文件目录下的话单文件
- 校验话单文件是否满足话单文件规范，不满足规范的文件自动丢弃移动到 error 目录下，不进行后续操作。
- 解压话单文件，读取话单文件内容，并且读取 excel 信息持久化到数据库内
- 持久化完成后，清理解压文件，并且将话单文件移动到backup 目录下，后续等待服务自动清理（本次不包含该功能）

每次话单分拣操作需要编排成一个流式任务，构建一个任务表，任务状态按步骤流转，每个步骤完成后流转到下一个步骤，全部完成后，将已完成任务移动到 任务表的备份表，持久化下来方便追溯，需要记录任务每个步骤的完成时间。
# 上下文信息

话单文件: 运营商通过上报话单文件到文件服务器，提供用户使用云服务的用量信息，运营平台通过话单文件具体的消费时间，消费用量，已经运营商计费规则，计算用户具体消费金额。

使用量单位：所有的使用量单位需要保存在一张数据库物理表中，类似
~~~
天 Day
月 Month
次数 Count
KB Kb
MB Mb
~~~

服务可用区：服务AZ，服务可用区也必须存储在数据库物理表中，表示服务提供的来源，类似北京beijin，亚洲asia

运营商：提供服务的供应商，上报话单信息的提供方，也需要保存在物理表上持久化。
# 约束以及一些要求
- 所有时间信息均使用 utc 时间以及格式
- 话单文件中存储英文格式，中文已经其他命名仅用于显示，在表数据以外的数据为异常数据
- 运营商名称需要与数据库中的物理表对应，在表数据以外的数据为异常数据
- 服务资源id统一 uuid 格式，全局唯一
- 分拣话单是一直进行的，所以设计上需要保证可靠性，保证可以中断后重新触发执行
- 分拣话单操作不是单线程的，多线程并发执行，并发度保证可以动态控制
- 单次分拣话单任务步骤时长不能过长，限制时长5min，超时自动重置任务
# 话单文件规范
- 话单文件上报周期最低是分钟级，最低周期是 30 分钟
- 话单文件命名以英文下划线分割，并且命名规则按顺序依次是
   运营商名_可用区_开始时间_结束时间_站点.zip
   例如：
~~~
ColoCloud_az1_202607120800_202607120930_asia.zip
~~~
- 一个话单文件 zip 包内，包含多个子话单 excel 文件，且必须满足命名规范
- 话单文件 zip 包内解压后格式必须为 csv 后缀的 excel 文件
- 话单文件内最多包含 100 个子文件，且总大小不能超过 100m
- 每个子话单文件 excel 文件内，最多 100 行，超过 100 行的文件数据自动丢弃
- 子话单文件excel 信息格式：
~~~
｜运营商名称｜服务 az｜服务资源 id｜开始时间戳｜结束时间戳｜使用量｜使用量单位｜
~~~
